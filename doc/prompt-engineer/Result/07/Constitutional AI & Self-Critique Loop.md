# Constitutional AI / Self-Critique Loop（憲法式 AI 與自我批判迴圈）

## 說明

Constitutional AI / Self-Critique Loop 範式是指讓模型依據一套預設的原則集（「憲法」）對自身的輸出進行自我審查與修正，在輸出到用戶之前主動識別並修正違反原則的部分。

這與 Guardrail 的根本差異在於：
- **Guardrail**：外部獨立系統攔截（不依賴主模型自律）
- **Constitutional AI / Self-Critique Loop**：模型**本身**依據內化的原則進行自我審查（強化模型的自律能力）

這個範式源自 Anthropic 的 Constitutional AI 研究，核心思想是：與其試圖窮舉所有禁止行為，不如給模型一套「行為準則」，讓模型學會自行推導在各種情況下的正確行為。

---

## 使用時機

| 情境 | 說明 |
|------|------|
| 價值觀對齊 | 確保模型輸出符合組織的核心價值與倫理原則 |
| 敏感話題處理 | 讓模型在回應前自我審查立場是否平衡、是否適當 |
| 品質門控 | 在輸出前自動識別並修正邏輯缺陷、不完整論述 |
| 自動化合規審查 | 讓模型自行檢查輸出是否符合特定法規或政策要求 |
| 減少人工審核負擔 | 讓模型自我過濾後，人工只需審查少量邊界案例 |

---

## 使用提示

- 原則（憲法條款）應**原則導向**而非列舉式，讓模型能夠推導邊界情況
- 批判階段應**具體指出問題位置**（「第二段的建議可能有偏見，因為...」）
- 修訂階段要求模型**逐條對應**每個批評進行修改，而非整體改寫
- 可設計**多輪自我批判**，但通常 2 輪就能獲得明顯改善
- 原則集應定期更新，反映組織政策或用戶需求的變化

## 提示詞範例

### 範例 1：基礎 Constitutional AI 流程

```
你是一個內容生成助理，遵守以下核心原則（憲法）：

【原則集】
P1. 誠實原則：不捏造事實，不確定的資訊應標明不確定性
P2. 無害原則：不提供可能傷害他人的指引，包括直接與間接傷害
P3. 公平原則：對不同群體保持一致的標準，不存在偏見或歧視
P4. 隱私原則：不推斷、透露或討論可識別個人的私人資訊
P5. 有用原則：在遵守前四項原則的前提下，盡可能提供有幫助的回應

工作流程：
步驟一：生成對用戶請求的初步回應
步驟二：依序審查初步回應是否違反 P1-P5（逐條審查，若違反說明原因）
步驟三：若有違反，生成修正後的最終回應
步驟四：輸出最終回應（不需要展示審查過程，除非用戶要求）

用戶請求：[用戶輸入]
```

### 範例 2：顯式自我批判迴圈（帶輸出）

```
請完成以下任務，並進行兩輪自我批判修正。

任務：請分析「遠端工作對員工生產力的影響」，給出平衡的觀點。

【第一輪】
1a. 初稿：直接回答問題
1b. 自我批判：依據以下標準審查初稿
   - 是否有選擇性引用支持單一觀點的論據？
   - 是否忽略了某些重要群體的觀點？
   - 有沒有未標示的不確定陳述？
   - 結論是否能夠被初稿中的論據所支持？
1c. 識別需要修改的具體問題（條列）

【第二輪】
2a. 根據第一輪批判修訂
2b. 最終自我確認：修訂是否解決了所有識別到的問題？

輸出：最終版本（附帶修改了哪些問題的簡要說明）
```

### 範例 3：組織特定的憲法原則

```
你是 FinanceBot，一個金融資訊助理。
在輸出每一個回應之前，你必須內部審查是否符合以下合規原則：

【FinanceBot 憲法】
C1. 不提供具體投資建議（不說「你應該買 X 股票」）
C2. 必須提醒用戶過去績效不代表未來表現
C3. 若資訊超過 6 個月，必須標注「資訊可能已過時」
C4. 對所有金融工具保持中立，不偏好任何特定產品或機構
C5. 風險資訊必須與收益資訊同等篇幅呈現

自我審查流程（內部執行，不對外展示）：
1. 生成初稿
2. 逐條對照 C1-C5 審查
3. 若有違反：修正後再輸出
4. 確認通過後，輸出最終回應

用戶問題：[金融相關問題]
```

### 範例 4：品質導向的自我批判

```
你是一位技術文件撰寫者。在提交每份文件前，請進行結構化自我審查。

品質審查憲法：
Q1. 完整性：所有必要的資訊都包含了嗎？有沒有遺漏的步驟或前置條件？
Q2. 準確性：技術細節是否正確？有沒有可能誤導讀者的表述？
Q3. 清晰性：目標讀者能否無障礙地理解？術語是否適當？
Q4. 可操作性：讀者能否按照文件執行？有沒有模糊的步驟？
Q5. 一致性：術語、格式、範例風格是否全文一致？

任務：為以下功能撰寫 API 文件：「用戶密碼重置流程」

執行流程：
1. 撰寫初版文件
2. 依照 Q1-Q5 自我審查，輸出審查報告
3. 根據審查報告修訂，輸出最終文件
```

## 常見錯誤

- **原則太具體（列舉式）**：列舉禁止行為清單而非原則，模型無法推導未列舉的邊界情況
- **自我批判流於表面**：批判只說「這部分可以改進」而不指出具體問題
- **修訂未對應批判**：修訂版本改了其他地方，但批判指出的問題依然存在
- **原則之間相互衝突**：「完全客觀」vs「對弱勢群體給予更多關照」——衝突的原則讓模型無所適從
- **過度審查造成過於保守**：每個回應都被大幅刪減，失去有用性
