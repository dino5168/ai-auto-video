# Output Formatter（輸出格式指定）

## 說明

Output Formatter 範式是指在提示詞中明確指定模型輸出的格式結構，包括 JSON、Markdown、表格、條列清單、程式碼區塊、純文字段落等。

明確的格式指定可以讓輸出直接被下游流程消費（如程式解析、文件渲染），避免因格式不一致造成的後處理成本。

---

## 使用時機

| 情境 | 說明 |
|------|------|
| 程式整合 | 輸出需要被程式碼直接解析（JSON、YAML、CSV） |
| 文件生成 | 輸出需要呈現為可讀文件（Markdown、表格） |
| 資料萃取 | 從非結構化文字中提取結構化資訊 |
| 報告產製 | 需要固定章節結構的分析報告 |
| 快速掃描 | 讀者需要快速瀏覽重點（條列、摘要框） |

---

## 使用提示

- 若格式複雜，提供格式範本或說明每個欄位的用途
- 明確指定不需要的元素（例如：「不要加入說明文字，只輸出 JSON」）
- 若有多種輸出區塊，說明各區塊的格式要求

---

## 提示詞範例

### 範例 1：輸出 JSON 格式

```
請分析以下產品評論，並以 JSON 格式輸出結果。
不要輸出任何其他文字，只輸出 JSON。

輸出格式：
{
  "sentiment": "positive" | "negative" | "neutral",
  "score": 0.0 ~ 1.0,
  "key_points": ["重點1", "重點2"],
  "suggested_action": "建議回應方式"
}

評論內容：
[貼上評論]
```

---

### 範例 2：Markdown 表格輸出

```
請比較以下三個雲端服務方案（AWS、GCP、Azure），
以 Markdown 表格格式輸出，包含以下欄位：
- 服務名稱
- 免費額度
- 計費方式
- 主要優勢
- 適用場景
```

---

### 範例 3：條列清單格式

```
請列出撰寫高品質提示詞的 10 個最佳實踐。
格式要求：
- 使用編號條列（1. 2. 3.）
- 每條包含：標題（粗體）+ 一句說明
- 不需要引言或結語
```

---

### 範例 4：多區塊混合格式

```
請針對以下程式碼進行 Code Review，輸出格式如下：

## 問題摘要
（一段話總結主要問題）

## 詳細問題列表
| 行號 | 問題類型 | 說明 | 建議修改 |
|------|----------|------|----------|

## 修改後程式碼
（程式碼區塊）

程式碼：
[貼上程式碼]
```

---

## 常見錯誤

- **格式說明模糊**：「用表格輸出」未指定欄位，導致表格結構不符預期
- **未禁止多餘內容**：模型在 JSON 前後加入說明文字，破壞可解析性
- **格式與內容衝突**：要求輸出條列但內容本質上是連續敘述，強制格式反而降低品質
