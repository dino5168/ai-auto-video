# Reflection（自我反思）

## 說明

Reflection 範式是指讓模型在給出初步答案後，回顧並審視自身的推論過程，主動找出潛在的錯誤、盲點、遺漏或邏輯缺陷，然後根據反思結果修正或補充答案。

這個範式的核心價值在於：模型的第一次輸出往往是「直覺回答」，透過強制反思步驟，可以捕捉到初稿中的錯誤，提升最終輸出的品質與可靠性。

---

## 使用時機

| 情境 | 說明 |
|------|------|
| 複雜推理問題 | 數學應用題、邏輯謎題、多步驟推導 |
| 重要決策分析 | 需要全面考量後果的建議或評估 |
| 程式碼審查 | 讓模型找出自己生成程式碼的潛在 Bug |
| 文章或文案審校 | 讓模型自行發現邏輯矛盾或表達問題 |
| 長篇輸出質量控制 | 確保長文的前後一致性 |

---

## 使用提示

- 明確分為「初稿」與「反思」兩個階段，讓模型不要跳過反思直接輸出最終答案
- 可指定反思的面向（邏輯一致性、事實正確性、完整性等）
- 可要求模型說明「反思後哪些地方需要修改、為何修改」
- 可迭代多輪反思（反思後再反思），但通常一輪已有明顯效益

## 提示詞範例

### 範例 1：數學推理反思

```
請解答以下問題，分兩個階段輸出：

**階段一：初步解答**
直接給出你的解題過程與答案。

**階段二：反思**
回顧你的解題步驟，檢查：
- 每一步的計算是否正確？
- 有沒有跳過某個必要的步驟？
- 假設前提是否合理？
如果發現錯誤，請給出修正後的答案。

問題：一個水箱可以裝 500 公升的水。一條水管每分鐘注入 12 公升，同時另一條排水管每分鐘排出 7 公升。若水箱目前是空的，需要多少分鐘才能裝滿？
```

---

### 範例 2：程式碼反思

```
請完成以下任務，分兩個階段輸出：

**階段一：寫出程式碼**
用 Python 寫一個函式，接受一個字串列表，回傳其中長度超過 5 的字串，並去除重複項目。

**階段二：自我審查**
審查你寫的程式碼，思考以下問題：
- 邏輯是否正確？用幾個測試案例（包含邊界情況）心算驗證
- 有沒有潛在的 Bug 或未處理的邊界情況？
- 效能是否有明顯問題？
- 命名與可讀性是否良好？

如果審查後發現問題，輸出修正版本並說明修改原因。
```

---

### 範例 3：建議方案反思

```
請針對以下情境提供建議，分兩個階段進行：

**階段一：初步建議**
給出你認為最佳的解決方案。

**階段二：反思與完善**
回顧你的建議，思考：
- 這個方案是否有潛在風險或副作用？
- 是否忽略了某些重要的利害關係人？
- 有什麼前提假設可能是錯的？
- 是否有更簡單或更低風險的替代方案？

基於反思，提供修正後的最終建議。

情境：我們的工程團隊生產力下降，主管希望引入每日進度追蹤系統來改善狀況。
```

## 常見錯誤

- **反思流於形式**：模型只是重複初稿內容，沒有真正找出問題
- **省略初稿**：要求太快到達答案，跳過反思思考過程
- **反思面向太廣泛**：未指定反思重點，模型反思內容零散無效
- **過度迭代**：對簡單任務強制多輪反思，浪費 token 且效益不高
