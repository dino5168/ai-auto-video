# Critique & Revise（批判與修訂）

## 說明

Critique & Revise 範式是指要求模型先產出一個初版答案，然後以批判的角度評估這份輸出的缺陷與不足，最後根據批評意見產出一個修訂後的更好版本。

與 Reflection 的差異在於：Reflection 側重於「找出推論過程的錯誤」，而 Critique & Revise 更像是「編輯審稿」——著重評估輸出品質的各個維度（清晰度、完整性、說服力、準確性等），並據此進行系統性改進。

---

## 使用時機

| 情境 | 說明 |
|------|------|
| 文案或文章撰寫 | 初稿後自動進行編輯優化 |
| 論述或說服性內容 | 找出論點的漏洞並加以補強 |
| 技術文件品質提升 | 評估說明是否清晰、完整、無歧義 |
| 回覆信件或溝通稿 | 確保語氣、內容符合情境需求 |
| 設計方案評估 | 找出方案的弱點並提出改善 |

---

## 使用提示

- 明確定義批判的標準（什麼叫做「好」的輸出）
- 批評階段要求模型產出具體的問題清單，而非模糊評語
- 修訂階段要求模型說明「針對哪個批評點做了哪些改動」
- 可指定批評的嚴格程度（溫和建議 vs. 嚴格審查）
- 可引入「以使用者視角」或「以專家視角」來進行批評

## 提示詞範例

### 範例 1：文案批判與修訂

```
請完成以下三個階段的任務：

**第一階段：撰寫初稿**
為以下產品撰寫一段 100 字的行銷文案。

產品：一款可折疊的輕量化筆記型電腦支架，重量 180g，可收納進筆電包側袋。

**第二階段：批評初稿**
以嚴格的行銷文案編輯身份，從以下角度批評你的初稿：
- 是否有清楚傳達核心價值主張？
- 目標受眾是否清晰？
- 情感共鳴是否足夠？
- 是否有任何冗餘或無效的用詞？
- 行動呼籲（CTA）是否明確？

請列出具體問題清單。

**第三階段：根據批評修訂**
根據上述批評，產出修訂後的文案，並說明針對每個問題點做了什麼改動。
```

---

### 範例 2：技術方案批判與修訂

```
請分三個階段完成：

**第一階段：提出方案**
為以下需求提出一個系統設計方案：
需求：一個支援每秒 1000 個請求的用戶通知系統，需要支援 Email、SMS、Push 三種通知管道。

**第二階段：批判方案**
以資深系統架構師的角度，批評你提出的方案：
- 單點失敗風險在哪裡？
- 擴展性的瓶頸在哪裡？
- 資料一致性如何保障？
- 有哪些遺漏的邊界情況？
列出至少 4 個具體問題。

**第三階段：修訂方案**
根據批評修訂方案，說明每個問題如何被解決。
```

---

### 範例 3：論述批判與修訂

```
請分三個階段回答：

**第一階段：初步論述**
論述：「遠端工作應該成為所有知識工作者的預設工作模式」。
請提出 3 個支持論點。

**第二階段：自我批判**
扮演一個反對這個立場的人，找出你的論點中：
- 邏輯上的漏洞
- 未考慮到的反例
- 過度簡化的假設
- 選擇性使用的證據

**第三階段：修訂論述**
整合批判意見，修訂你的論述，使其更加嚴謹、平衡且有說服力。
```

## 常見錯誤

- **批評過於表面**：只說「可以更好」而沒有指出具體問題
- **修訂未對應批評**：修訂版本與批評內容脫節，沒有逐一回應
- **批評標準不明確**：沒有告訴模型從哪些維度進行批評
- **跳過初稿直接輸出「完美版」**：失去批判思考過程的價值
