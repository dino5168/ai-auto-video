# Cognitive Verifier（認知驗證器）

## 說明

Cognitive Verifier 範式是指將一個複雜問題主動拆解為多個可獨立驗證的子問題，分別推導出每個子問題的答案，最後整合所有子答案得出最終結論。

這個範式的核心在於「分而治之」——當一個問題太複雜而難以直接正確回答時，透過降低每個子問題的難度，並在整合前驗證每一步，可以大幅降低單步錯誤傳播到最終答案的風險。

與 Chain-of-Thought 的差異：CoT 是沿著一條推理線前進；Cognitive Verifier 是同時維護多條獨立推理線，最後交叉整合。

---

## 使用時機

| 情境 | 說明 |
|------|------|
| 多維度問題評估 | 問題涉及多個相互獨立的評估維度 |
| 複合條件判斷 | 最終結論取決於多個條件是否同時成立 |
| 風險評估 | 需要從技術、法律、財務等多個面向獨立評估 |
| 資訊整合分析 | 多個資訊來源需要分別驗證後再交叉比對 |
| 複雜程式邏輯驗證 | 分別驗證各個模組或條件後再整體評估 |

---

## 使用提示

- 明確列出需要分別驗證的子問題（或要求模型自行拆解）
- 要求每個子問題給出明確的結論（是/否、數值、評級）再進入整合階段
- 整合階段要說明各子問題結論如何影響最終答案
- 若子問題之間有依賴關係，須先處理前置子問題

## 提示詞範例

### 範例 1：投資決策多維驗證

```
請評估以下投資決策的可行性。
先將問題拆解為獨立子問題，分別驗證每個子問題，最後整合結論。

決策：投資一家剛成立兩年的 B2B SaaS 新創公司，金額為 500 萬台幣。

請分別驗證以下子問題（每題給出明確結論）：

子問題 1：市場規模是否足夠支撐這個商業模式？
子問題 2：產品的技術壁壘是否足夠高？
子問題 3：目前的財務狀況是否健康？
子問題 4：創辦團隊的背景是否匹配這個市場？
子問題 5：500 萬的估值是否合理？

整合：基於以上五個子問題的結論，給出最終投資建議。
```

### 範例 2：合約條款合規驗證

```
請驗證以下合約條款是否符合台灣法律要求。
將驗證工作拆解為獨立的法律子問題，分別驗證後整合結論。

合約條款：
「乙方同意在合約終止後三年內，不得在台灣境內從事任何與甲方業務相似的工作，
違反者須賠償甲方新台幣一百萬元。」

子問題 1：競業禁止期間（三年）是否符合勞動基準法的合理限制？
子問題 2：地理範圍限制（台灣境內）是否合理？
子問題 3：「業務相似」的模糊性是否構成法律風險？
子問題 4：違約金金額（100 萬）是否可能被法院調整？
子問題 5：整體條款是否需要補充任何必要要素才能生效？

整合：這個條款目前的風險等級為何？需要哪些修改？
```

### 範例 3：系統效能問題診斷

```
我們的 API 回應時間從平均 200ms 上升到 800ms，請協助診斷。

請分別驗證以下可能的原因（每個給出「可能」/「不太可能」的初步判斷與理由）：

子問題 1：資料庫查詢是否有效能問題？（請問我哪些資訊才能判斷）
子問題 2：是否有 N+1 查詢問題？
子問題 3：快取層是否失效或命中率下降？
子問題 4：是否有記憶體洩漏導致 GC 壓力增加？
子問題 5：是否是第三方 API 呼叫的延遲造成的？
子問題 6：是否是流量增加導致的資源瓶頸？

整合：依照可能性高低排序，給出診斷優先順序與對應的調查方法。
```

## 常見錯誤

- **子問題之間有隱性依賴**：沒有按照依賴順序處理，導致後面的子問題結論建立在未驗證的假設上
- **整合階段忽略子問題結論**：最終結論與各子問題的結論不一致或矛盾
- **子問題切割不當**：子問題互相重疊或遺漏重要面向
- **把 CoT 誤用為 Cognitive Verifier**：用線性推理代替多線並行驗證，失去交叉驗證的優勢
