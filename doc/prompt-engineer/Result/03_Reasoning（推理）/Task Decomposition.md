# Task Decomposition（任務拆解）

## 說明

Task Decomposition 範式是指要求模型主動分析問題的結構，並自主將其拆解為可獨立處理的子任務，然後有系統地逐一解決。

與 Plan-Then-Execute 的關鍵差異：
- **Plan-Then-Execute**：使用者明確知道任務的大致步驟，要求模型制定計畫後等待確認再執行
- **Task Decomposition**：**拆解本身由模型發起**——模型需要理解問題複雜性後，自行決定如何分解，不需要使用者預先定義步驟

---

## 使用時機

| 情境 | 說明 |
|------|------|
| 問題結構模糊 | 使用者知道目標但不清楚如何分解步驟 |
| 需要模型展現問題分析能力 | 評估模型對問題複雜性的理解 |
| 開放性複雜任務 | 任務有多種可能的分解方式，讓模型選擇最適合的 |
| 學習與教學場景 | 透過拆解過程幫助使用者理解問題的結構 |
| Agent 自主執行情境 | 模型需要獨立判斷如何拆解並執行長任務 |

---

## 使用提示

- 要求模型先輸出拆解結構再執行，方便使用者理解與確認
- 可要求模型說明拆解的依據（為何這樣分組）
- 允許模型在執行過程中動態調整拆解方式（若發現原始拆解不合適）
- 可要求識別子任務之間的依賴關係與執行順序

## 提示詞範例

### 範例 1：開放性技術問題的自主拆解

```
我想建立一個個人品牌網站，展示我的作品集、部落格文章與聯絡資訊。
我有基本的 HTML/CSS 知識，但沒有用過任何框架。

請你：
1. 自主分析這個任務的複雜度，並拆解為子任務
2. 說明你的拆解依據
3. 標示每個子任務的預估難度與依賴關係
4. 按照優先順序列出執行序列

完成拆解後，從第一個子任務開始逐步協助我執行。
```

---

### 範例 2：商業問題的結構化拆解

```
我們的電商平台訂單轉換率在過去一個月下降了 15%，
請幫我拆解這個問題並制定診斷方案。

要求：
- 自主識別可能影響轉換率的所有面向
- 將問題拆解為可獨立調查的子問題
- 為每個子問題說明：需要什麼資料才能診斷？
- 依照「最可能是原因」的機率排序子問題
- 提出一個系統性的調查計畫
```

---

### 範例 3：學習計畫的自主拆解

```
我是一位有 3 年 Java 後端經驗的工程師，
想在三個月內學會用 React 開發前端應用。

請幫我：
1. 分析學習 React 的知識結構，識別核心子領域
2. 評估我的 Java 背景哪些知識可以遷移，哪些需要從頭建立
3. 將學習任務拆解為每週可執行的里程碑
4. 識別最關鍵的學習瓶頸點

請先輸出完整的拆解結構，再詳細說明第一個月的學習路徑。
```

---

### 範例 4：程式重構的任務拆解

```
以下是一個 500 行的 Python 腳本，原本是一個臨時的資料處理工具，
現在需要重構為一個可維護、可測試的模組。

請自主分析：
1. 這個重構任務可以如何拆解？
2. 哪些部分應該優先處理？
3. 哪些改動之間有依賴關係？
4. 預計每個子任務的影響範圍

輸出你的拆解方案後，等待我確認再開始執行。

程式碼：
[貼上程式碼]
```

## 常見錯誤

- **拆解粒度不當**：子任務太大（等於沒拆）或太小（過度碎片化）
- **忽略依賴關係**：拆解後沒有說明執行順序，導致後續步驟的前置條件未完成
- **拆解後不執行**：只輸出拆解結構而不繼續推進，失去範式的完整價值
- **與 Plan-Then-Execute 混淆**：使用者已知步驟時應使用 Plan-Then-Execute，只有問題結構模糊時才需要 Task Decomposition
