# Plan-Then-Execute（先規劃後執行）

## 說明

Plan-Then-Execute 範式是指讓模型在實際執行任務之前，先輸出一份明確的執行計畫，等待使用者確認或調整後，再按照計畫逐步執行。

這個範式將「思考怎麼做」與「實際去做」分為兩個階段，讓使用者能夠在投入大量計算資源前介入修正方向，避免走錯路後才發現需要重來。適用於任何多步驟、高成本、難以回滾的任務。

---

## 使用時機

| 情境 | 說明 |
|------|------|
| 多步驟程式開發 | 在寫程式前先確認架構設計與實作路徑 |
| 資料處理 Pipeline | 確認轉換邏輯正確後再執行，避免資料損壞 |
| 複雜文件撰寫 | 先確認大綱與章節結構，再逐段展開 |
| 系統重構 | 確認重構策略與影響範圍後再動工 |
| 研究或分析任務 | 先確認分析框架與資料來源，再進行分析 |

---

## 使用提示

- 規劃階段要求輸出具體可執行的步驟清單，而非模糊描述
- 每個計畫步驟應包含：做什麼、依賴什麼、預期產出
- 明確告知模型「等待確認後再執行」，避免模型直接跳到執行
- 執行階段可要求每完成一個步驟後確認，而不是一次性輸出所有結果

## 提示詞範例

### 範例 1：功能開發計畫確認

```
我需要在現有的 Express.js 專案中新增用戶認證功能（JWT）。
請先輸出你的實作計畫，不要開始寫程式碼。

計畫格式：
1. 需要安裝的套件
2. 需要新增/修改的檔案清單
3. 每個步驟的實作內容與預期結果
4. 可能的風險點與注意事項

等我確認計畫無誤後，再逐步執行。
```

---

### 範例 2：長篇文章規劃

```
我需要撰寫一篇 3000 字的技術文章，主題是「RAG（Retrieval-Augmented Generation）
在企業知識庫應用的最佳實踐」。

請先輸出：
1. 文章結構與各章節標題
2. 每個章節的核心論點（2~3 句話）
3. 預計每個章節的字數分配
4. 你計畫引用的主要概念或案例類型

等我確認結構後，再開始撰寫。
```

---

### 範例 3：資料庫遷移計畫

```
我需要將以下 MySQL 資料表結構遷移，新增 `created_at` 和 `updated_at` 欄位，
並為 `email` 欄位建立唯一索引。

目前的資料表有約 50 萬筆資料，系統需要保持 24/7 可用。

請先輸出遷移計畫，包含：
1. 遷移步驟的詳細說明
2. 每個步驟的預計影響與風險
3. 回滾方案
4. 對系統可用性的影響評估

不要輸出任何 SQL，等我確認計畫後再執行。
```

---

### 範例 4：研究分析框架確認

```
請幫我分析「混合辦公政策對員工生產力的影響」這個議題。

在開始分析前，請先輸出你的分析框架：
1. 你打算從哪些維度分析這個問題？
2. 你會使用哪些衡量標準或指標？
3. 你計畫如何處理不同研究結果之間的矛盾？
4. 分析的範圍限制與假設前提

等我確認框架後，再進行完整分析。
```

## 常見錯誤

- **計畫過於粗糙**：計畫只有模糊的方向描述，無法讓使用者做出有意義的評估
- **模型不等確認就執行**：提示詞未明確要求「等待確認」，模型直接開始執行
- **計畫與執行脫節**：執行時偏離了確認的計畫，失去先規劃的意義
- **對簡單任務過度規劃**：增加往返次數但沒有降低風險，適得其反
