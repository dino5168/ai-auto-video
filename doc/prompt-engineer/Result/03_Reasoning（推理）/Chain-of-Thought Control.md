# Chain-of-Thought Control（推理鏈控制）

## 說明

Chain-of-Thought Control 範式是指在提示詞中明確控制模型推理鏈的展開方式，包括：要求逐步推導、使用零樣本 CoT 觸發語句、強制模型輸出中間推理步驟，或限制推理的深度與廣度。

CoT 的核心價值在於：讓模型「邊想邊寫」而非直接跳到結論，這能顯著提升複雜推理任務的準確性，因為中間步驟會相互約束，降低跳躍性錯誤的機率。

---

## 使用時機

| 情境 | 說明 |
|------|------|
| 多步驟數學或邏輯題 | 需要逐步推導才能確保答案正確 |
| 複雜因果分析 | 釐清多個因素之間的連鎖關係 |
| 程式除錯 | 逐步追蹤執行流程找出問題所在 |
| 法律或合約解讀 | 逐條分析條款間的邏輯關係 |
| 需要可審計的推理過程 | 使用者需要驗證模型的每一步推論 |

---

## 使用提示

- **零樣本 CoT**：在提示末尾加上「讓我們一步一步思考」即可觸發推理鏈，無需示例
- **強制中間步驟**：明確列出推理必須經過的檢查點或階段
- **分離思考與輸出**：讓模型先在 `<thinking>` 區塊推理，再輸出簡潔的最終答案
- **限制推理深度**：對簡單問題可省略 CoT，避免過度複雜化

## 提示詞範例

### 範例 1：零樣本 CoT 觸發

```
一個工廠每天生產 240 個零件，其中 5% 為不良品。
品管部門每天能檢查 80% 的產品，並能找出其中 90% 的不良品。
請問每天平均有多少不良品會流出工廠？

讓我們一步一步思考。
```

---

### 範例 2：強制中間步驟

```
請分析以下商業決策的可行性，必須依序完成每個步驟，
並在進入下一步前完整說明當前步驟的結論：

步驟一：釐清決策的目標與成功標準
步驟二：識別主要的假設前提
步驟三：分析支持這個決策的論據
步驟四：分析反對這個決策的論據
步驟五：評估每個假設前提失效的影響
步驟六：綜合以上給出最終建議

決策：將客服部門的 60% 工作量轉移給 AI 聊天機器人處理。
```

---

### 範例 3：分離思考與最終答案

```
請回答以下邏輯題。
先在 <thinking> 標籤內完整展開你的推理過程，
然後在 <answer> 標籤內只給出簡潔的最終答案。

題目：
A、B、C 三個盒子，其中一個裝著金幣，另外兩個是空的。
每個盒子上有一句話：
- A：「金幣不在 B 裡」
- B：「金幣在 A 裡」
- C：「金幣不在 A 裡」
已知只有一句話是真的，金幣在哪個盒子裡？
```

---

### 範例 4：程式除錯的逐步追蹤

```
以下 Python 函式有一個 Bug，請逐步追蹤執行流程找出問題：

步驟一：描述函式的預期行為
步驟二：用具體輸入（例如 [3, 1, 4, 1, 5]）模擬每行程式碼的執行狀態
步驟三：找出實際行為與預期行為出現分歧的位置
步驟四：說明 Bug 的根本原因
步驟五：提出修正方案

程式碼：
def find_second_largest(nums):
    nums.sort()
    return nums[-2]
```

## 常見錯誤

- **對簡單問題強制 CoT**：增加 token 消耗但效益為零，甚至讓模型在推理中引入不必要的複雜性
- **CoT 步驟定義模糊**：要求「一步一步」但未定義步驟的粒度，導致推理過於粗糙
- **跳過中間驗證**：CoT 只是陳述步驟而非真正驗算，失去推理鏈的糾錯功能
- **推理與答案混合**：沒有區分推理過程與最終答案，讓輸出難以閱讀與使用
