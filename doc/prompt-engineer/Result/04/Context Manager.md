# Context Manager（上下文管理）

## 說明

Context Manager 範式是指明確控制放入 context window 的內容，包括：壓縮冗餘資訊、截斷過長的歷史對話、將長文摘要後再注入、以及依照重要性排列內容的優先順序。

由於語言模型的 context window 有有限的長度，且模型對於 context 開頭與結尾的注意力通常優於中段（Lost in the Middle 效應），因此如何組織與管理注入的資訊，直接影響輸出品質與可靠性。

---

## 使用時機

| 情境 | 說明 |
|------|------|
| 長對話歷史管理 | 多輪對話累積後 context 超載，需要摘要壓縮舊輪次 |
| 大型文件處理 | 文件長度超過 context 上限，需要截斷或摘要後注入 |
| 多文件資訊整合 | 多份文件同時注入時，需要排序優先級 |
| Agent 工具回傳值管理 | 工具呼叫返回大量資料，需要提取關鍵部分再傳遞 |
| 成本控制 | 減少不必要的 token 消耗，降低 API 費用 |

---

## 使用提示

- 將最關鍵的資訊放在 context 的**開頭或結尾**，避免放在長段落的中間
- 對歷史對話進行**滾動摘要**（Rolling Summary）：保留摘要 + 最近幾輪完整對話
- 明確告知模型哪些資訊是**高優先級**（必須考量）與**低優先級**（參考用）
- 使用結構化標籤（如 `<context>`、`<background>`）幫助模型區分不同類型的 context

## 提示詞範例

### 範例 1：對話摘要壓縮

```
以下是我們之前對話的摘要，以及最近兩輪的完整對話記錄。
請基於這些資訊繼續協助我。

<conversation_summary>
我們正在設計一個電商平台的訂單管理系統。
已確認的技術選型：Node.js + PostgreSQL + Redis。
已完成：資料庫 Schema 設計、用戶認證模組。
待完成：訂單狀態機設計、通知系統整合。
主要限制：系統需支援每秒 500 筆訂單的峰值處理量。
</conversation_summary>

<recent_messages>
[用戶]: 訂單狀態機需要支援哪些狀態？
[助手]: 建議包含：pending → confirmed → processing → shipped → delivered / cancelled / refunded...
[用戶]: 如果訂單在 processing 狀態時要取消，需要觸發退款，這個邏輯應該放在哪裡？
</recent_messages>

請繼續回答最後這個問題。
```

### 範例 2：長文件分段注入與優先級標示

```
以下是分析所需的參考資料，依優先級分類：

<high_priority>
【財報摘要】Q3 2025 營收 3.2 億，較去年同期成長 18%，但毛利率下降 3 個百分點至 42%。
主要成本壓力來源：原物料上漲 15%、人力成本上漲 8%。
</high_priority>

<medium_priority>
【市場背景】同業平均毛利率為 45%，行業領導者為 52%。
電商滲透率在本季提升至 38%，較上季成長 5 個百分點。
</medium_priority>

<low_priority>
【歷史數據】過去五年平均毛利率為 44.5%，最高為 2022 年 Q1 的 48%。
</low_priority>

請基於以上資料，分析本季毛利率下降的主要原因，並給出改善建議。
高優先級資料為主要分析依據，其他資料作為背景參考。
```

## 常見錯誤

- **未壓縮舊對話就堆疊新輸入**：導致 context 快速膨脹，且舊資訊占據重要位置
- **重要資訊放在中段**：模型對 context 中間部分的注意力較低（Lost in the Middle）
- **注入過多低相關性內容**：稀釋模型對關鍵資訊的注意力
- **未標示資訊優先順序**：模型無法判斷哪些資訊更重要，可能平等對待所有內容
