# RAG（檢索增強生成）

## 說明

RAG（Retrieval-Augmented Generation）是指在生成答案之前，先從向量資料庫、搜尋引擎或其他知識庫中檢索與問題最相關的文本片段（chunks），再將這些片段注入 prompt，讓模型基於檢索到的內容生成有依據的回答。

RAG 解決了 LLM 的兩個核心限制：訓練知識截止日期的限制，以及無法存取私有或領域專屬知識庫的限制。透過 RAG，模型的「知識」可以動態更新，且可追溯到具體來源。

---

## 使用時機

| 情境 | 說明 |
|------|------|
| 企業知識庫問答 | 基於公司文件、規範、SOP 回答員工問題 |
| 客服自動化 | 從產品手冊、FAQ 中檢索相關內容回答客戶 |
| 法規與合規查詢 | 從法律條文、內部政策中找到適用依據 |
| 技術文件輔助 | 基於 API 文件、架構文件回答開發問題 |
| 研究資料整合 | 從論文資料庫中檢索相關研究後生成摘要 |

---

## 使用提示

- 檢索品質決定生成品質：**chunk 的切割策略**（大小、重疊度）至關重要
- 在 prompt 中明確要求模型**只基於提供的文件回答**，若找不到答案要明確說明
- 為每個檢索片段標注**來源資訊**（文件名稱、頁碼、章節），方便引用與查證
- 設定**相關性閾值**：低於閾值的檢索結果不應注入，以免引入干擾資訊
- 考慮使用**混合檢索**（向量搜尋 + 關鍵字搜尋）提升召回率

## 提示詞範例

### 範例 1：基礎 RAG 問答框架

```
你是一個知識庫問答助手。請嚴格根據以下檢索到的文件片段回答用戶的問題。

規則：
1. 只使用提供的文件內容作為回答依據
2. 若文件中沒有足夠資訊，回答「根據現有文件，無法回答此問題，建議聯繫 [部門名稱]」
3. 回答時標注資訊來自哪份文件（使用文件 ID）
4. 不要加入文件之外的推測或補充

<retrieved_chunks>
[chunk_id: DOC-001, source: 產品規格書 v2.1, page: 12]
本產品最大承重為 150 公斤，建議使用承重不超過 120 公斤以確保安全邊際。
適用環境溫度：-10°C 至 50°C。

[chunk_id: DOC-002, source: 安全使用手冊, page: 3]
請勿在潮濕環境中使用本產品。若產品接觸液體，應立即斷電並聯繫售後服務。
</retrieved_chunks>

用戶問題：這個產品可以在浴室使用嗎？最大可以承重多少？
```

### 範例 2：帶引用格式的 RAG

```
你是一個法規查詢助手。請根據以下檢索到的法規條文回答問題，
並在回答中以 [來源: 文件名稱, 條號] 的格式標注引用依據。

<retrieved_chunks>
[source: 個人資料保護法, 第5條]
個人資料之蒐集、處理或利用，應尊重當事人之權益，依誠實及信用方法為之，
不得逾越特定目的之必要範圍，並應與蒐集之目的具有正當合理之關聯。

[source: 個人資料保護法, 第19條]
非公務機關對個人資料之蒐集或處理，應有特定目的，並符合下列情形之一：
一、法律明文規定。
二、與當事人有契約或類似契約之關係...
</retrieved_chunks>

問題：企業蒐集客戶個人資料時，有哪些法律要求？
```

### 範例 3：多輪 RAG 對話的 context 管理

```
你是一個基於公司知識庫的問答助手。

以下是本輪問題最相關的檢索結果（相關性分數 > 0.75）：

<retrieved_chunks relevance_threshold="0.75">
[chunk_id: KB-234, score: 0.92, source: 員工福利手冊 2025, section: 4.2]
員工子女教育補助：每年最高補助新台幣 30,000 元，需於每年 9 月 30 日前申請。
補助範圍：幼稚園至大學學費，不含研究所。

[chunk_id: KB-235, score: 0.88, source: 員工福利手冊 2025, section: 4.3]
申請流程：填寫「子女教育補助申請表」，附上就讀證明及學費收據，送交人資部門審核。
審核時間約 2 週，補助款於審核通過後下月薪資一同發放。
</retrieved_chunks>

<low_relevance_notice>
以下相關性低於閾值的結果未注入，若需要可另行查詢：KB-301 (score: 0.43)
</low_relevance_notice>

員工問題：我的孩子讀大學，請問教育補助怎麼申請，最多可以拿多少？
```

## 常見錯誤

- **chunk 切割過大**：單個 chunk 包含過多不相關內容，稀釋相關資訊的比例
- **chunk 切割過小**：缺乏上下文，模型難以理解片段含義
- **未設相關性閾值**：將低相關性結果也注入，引入干擾甚至矛盾資訊
- **未要求模型標注來源**：答案無法追溯到具體文件，失去 RAG 的可信度優勢
- **依賴 RAG 替代訓練知識**：對於通用知識問題，RAG 反而增加延遲與成本
